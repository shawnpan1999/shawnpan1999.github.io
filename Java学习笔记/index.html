关键字 static、final 和 常量

static

静态变量

  • 无需定义对象，是直接依赖于类而存在的
  • 通过实例对象更改静态变量的值会影响类和所有对象的变量值（即他们在内存中是同一个空间）

静态方法

  • 无需依赖对象，可以通过类名直接引用（最好只用类名来访问，用对象访问会出现警告）
  • 静态方法中不能直接使用非静态变量
  • 静态方法中不能直接引用非静态方法

静态代码块（常使代码混乱，不建议使用）

  • 静态代码块只在类第一次加载的时候调用
  • 执行顺序：静态代码块 -> 匿名代码块 -> 构造函数

final

final 类

  • final 类不能被继承

final 方法

  • final 方法不能被子类重写（但可以重载）

final 变量

  • final 变量不能被再次赋值

拼接 —— + 号和 append

对于下面这段代码

用 + 号进行拼接消耗的时间比 append 方法多出非常多。

+ 号的实质：对于 str += "a" ，编译器会将其处理为 new StringBuilder().append(str).append(“a”); 。

这就导致了，在循环中对字符串进行 + 拼接，会一直生成新的 StringBuilder 对象，导致效率非常低。

这个例子印证了 String 是个不可变对象，对其进行的所谓的”修改“，实质上是另外 new 一个 StringBuilder 出来，而原先字符串的空间仍被占用，等待回收。

结论：在非循环体内使用 + 和 append 方法拼接字符串，差别不大；在循环中，+ 号拼接非常影响效率。养成使用 append 拼接字符串的习惯是很有必要的。

package 和 import

package 包

类似于 C++ 里的 namespace 。

目的

使得多个 java 类文件不必放在同一个目录下面，而是支持跨目录放置和调用 java 类。

包名

通常使用逆序的域名作为包名（确保唯一性、范围从大到小），如 cn.edu.nuaa 。

包名和目录层次一样，如 cn.edu.nuaa 下的类必须放在 cn\edu\nuaa\ 目录下

类的完整名称：包名 + 类名，如 cn.edu.nuaa.ClassName 。

import 引用

类似于 C++ 里的 include 。

正常在使用其他包里的类时，需要使用全名，如 new cn.edu.nuaa.ClassName() ；为了方便，我们可以把 cn.edu.nuaa 引用，之后就可以直接 new ClassName() 了。

写法

  • import cn.edu.nuaa.ClassName 精确引用该目录下的 ClassName 类
  • import cn.edu.nuaa.* 引用该目录下的所有类（不包括子文件夹，如 import cn.* 是不行的）
  • 如果两个类在同一目录下，则无需 import ，直接使用即可

try-catch-finally 捕获异常

结构

通常使用 try-catch-finally 的保护机制（try 一定要有，catch 和 finally 至少有一个）来处理异常。

  • try：正常执行的业务逻辑代码
  • catch：当 try 中捕获到异常后，会跳转至 catch 部分匹配异常。（如无异常则不会 catch ）
  • finally：try 和 catch 部分执行完毕后，最后都会执行的部分。（即使是 catch 部分自己也含有异常，那么将在 finally 执行完毕后才报出前面 catch 的异常。

List 列表

  • 是一个有序的 Collection
  • 允许重复元素
  • 可以嵌套

List 中元素的编号是从 0 开始的。

List 接口的子类主要有三个：ArrayList、LinkedList、Vector：

ArrayList

本质上是数组实现的列表，区别是 ArrayList 的容量是动态的，可以随时扩充。

优点

  • 可随时扩充 (元素填满时甚至会自动扩充容器大小的 50%)
  • 可用索引位置 (如 a[1]) 快速访问指定位置元素

缺点

  • 不支持同步
  • 指定位置的插入、删除操作代价很大，执行起来需要挪动一大片数据

适合场景

  • 适合存放不太需要变动，主要用于被查询的数据

常用方法

add(a) 在末尾增加元素 a

add(x, a) 在下标为 x 的地方插入元素 a ，后面的元素向后挪动

get(x) 访问下标为 x 的元素

remove(x) 移除下标为 x 的元素，后面的元素往前挪动

LinkedList

本质上是双向链表实现的列表

优点

  • 插入、删除操作容易

缺点

  • 不支持同步
  • 访问指定位置元素较困难

适合场景

  • 用于经常变化的数据

常用方法

add(a) 在末尾增加元素 a

add(x, a) 在 x+1 的地方插入元素 a

addFirst(a) 在头部增加元素 a

get(x) 访问 x+1 个元素

remove(x) 移除第 x+1 个元素

当需要遍历 LinkedList 时，最好不要用 get() 随机访问的方法去访问每一个元素，这样花费的代价太大 (花费时间约是下两种方法的一千倍)。

建议采用迭代器 Iterator 或者 for 循环的方式，从头到尾访问。

信息共享

多线程工作时一般通过共享变量来达到信息共享的目的。

例如，现在我们有一百张电影票待售，而程序允许同时有多个线程进行购票操作，让这些线程使用的是同一个内存中的值 (剩余票数)，就是共享变量。

简单信息共享的缺陷

直观地看来，我们可以简单地通过类中 static 变量来达到共享变量的目的，然而这样做有着致命缺陷：

SellTickets.java

运行结果：

可以看到，这些票并不是按顺序一张一张卖出的，其中，编号 100 的票被卖出去了三次……

出现这样的缺陷的原因是工作缓存副本的机制和关键步骤未加锁。

工作缓存副本：

卖票操作 SellTickets.tickets-- 不是一个原子性的操作 (如同原子一般不可再分的操作，例子中的自减操作仍是由多个步骤组成的) 。

这样的自减操作可以分为：

  1. 复制一份主存(正本)，放到工作缓存(副本)中
  2. CPU对副本的数据进行运算 (减1)
  3. CPU 将运算后的结果写入到副本中
  4. 将副本的数据更新到正本中

在串行程序下，这样的操作步骤没有任何问题，然而一旦多个线程同时复制这个正本到其各自的工作缓存中，就会出现有的线程已经执行自减操作，而另一个线程的副本还是旧值。这也就是上述例子中 100 编号的票被卖出去三次的原因。

volatile 保证新值对其他线程的可见性

采用 volatile 修饰我们的变量，可以使得变量在一个工作缓存中被修改之后，其他的线程能够马上得到最新的值。

synchronized 给方法/代码块加锁

对于增减票数这种关键步骤，如果没有加以限制，会出现多个线程同时运行这一步骤的情况。

如果能够使某一个线程运行这部分代码时，其他线程不能同时运行这段代码，这就叫做互斥。

如果在互斥的基础上，再满足这些线程按照特定的先后顺序运行这部分代码，这叫做同步。

就我们的例子而言，若能实现互斥，那么就可以避免多个线程同时购票的情况。于是我们可以使用 synchronized 关键字来修饰方法或者代码块，使得这个方法/代码块不能同时被多个线程运行。

手动管理多线程

比起让各个线程去“抢”CPU，我们更倾向于让线程按照我们规定的顺序、动作来执行，这就涉及到 Java 给我们的多线程管理的 API 。

线程状态

  • NEW 刚创建 (new)
  • RUNNABLE 就绪 (start 后)
  • RUNNING 运行中 (run 执行时)
  • BLOCK 阻塞 (人为地 sleep)
  • TERMINATED 结束

API

  • sleep 阻塞(休眠)一定时间
  • wait/notify/notifyAll 休眠，并等待别人唤醒
  • join 排到某个线程后面，等该线程结束就运行
  • interrupt 向另外一个线程发送中断信号，该线程可以捕获 InterruptedException 并进行下一步处理
  • setDaemon(true) 设置为守护线程 (现在会随着 main 的结束而结束)
